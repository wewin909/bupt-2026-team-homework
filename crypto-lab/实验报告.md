## 二、实验原理

### 2.1 对称加密 —— AES（Advanced Encryption Standard）

#### 2.1.1 算法思想

AES 是美国国家标准与技术研究院（NIST）于 2001 年发布的分组对称加密标准，用于取代 DES。其核心是基于**代换-置换网络（SPN）**结构的迭代分组密码：

- **明文分组长度**：128 bits（16 bytes），固定不变
- **密钥长度**：128 / 192 / 256 bits 三种规格（本实验选用 256 bits）
- **轮数**：与密钥长度相关 —— AES-128 为 10 轮，AES-192 为 12 轮，AES-256 为 14 轮

每轮操作包括四个步骤：
1. **SubBytes**（字节代换）：通过 S 盒对每个字节进行非线性替换，提供**混淆**（Confusion）
2. **ShiftRows**（行移位）：对状态矩阵各行循环左移不同偏移量，提供**扩散**（Diffusion）
3. **MixColumns**（列混合）：在有限域 GF(2⁸) 上进行矩阵乘法，进一步增强扩散（最后一轮省略）
4. **AddRoundKey**（轮密钥加）：将轮密钥与状态矩阵逐字节异或

**AES 解密**是加密的逆过程，使用相同的轮密钥但**逆序**执行：InvShiftRows → InvSubBytes → AddRoundKey → InvMixColumns。

> 注意：本实验不要求手写 AES 内部轮函数，使用成熟的 `cryptography` 库，重点在于理解整体流程、分组模式选择与参数配置。

#### 2.1.2 分组模式 —— CBC（Cipher Block Chaining）

ECB（电子密码本）模式下相同明文块产生相同密文块，存在严重安全隐患。CBC 模式引入 **IV（初始化向量）**，使每个明文块在加密前先与前一个密文块异或：

$$C_i = E_K(P_i \oplus C_{i-1}), \quad C_0 = IV$$

- 同一明文在不同 IV 下产生完全不同密文
- 解密需使用加密时相同的 IV：$P_i = D_K(C_i) \oplus C_{i-1}$
- 填充方案：PKCS7（缺几个字节就补几个值为"缺少数"的字节）

**密钥长度选择理由**：AES-256 提供 256 bits 安全强度，可抵抗量子计算加速的 Grover 算法（将暴力搜索从 $O(2^{256})$ 降为 $O(2^{128})$），满足长期安全需求。

#### 2.1.3 算法步骤（AES-CBC 加解密）

```
加密流程:
  输入: 明文 Plaintext, 密钥 K (256 bits)
  输出: (IV, Ciphertext)

  1. 生成随机 128-bit IV
  2. 对明文进行 PKCS7 填充，使长度为 128 bits 的整数倍
  3. 将填充后数据分为 n 个 128-bit 分组: P₁, P₂, ..., Pₙ
  4. C₀ = IV
  5. For i = 1 to n:
       Cᵢ = AES_Encrypt(K, Pᵢ ⊕ Cᵢ₋₁)
  6. 返回 (IV, C₁ || C₂ || ... || Cₙ)

解密流程:
  输入: (IV, Ciphertext), 密钥 K
  输出: 明文 Plaintext

  1. 将密文分为 n 个 128-bit 分组: C₁, C₂, ..., Cₙ
  2. C₀ = IV
  3. For i = 1 to n:
       Pᵢ = AES_Decrypt(K, Cᵢ) ⊕ Cᵢ₋₁
  4. 去除 PKCS7 填充
  5. 返回 明文
```

#### 2.1.4 算法复杂度

| 指标 | AES-256 |
|------|---------|
| 密钥空间 | $2^{256}$ |
| 加密时间复杂度 | O(n)，n 为明文分组数 |
| 解密时间复杂度 | O(n)，与加密对称 |
| 空间复杂度 | O(1)（除输入输出外几乎不需额外空间） |
| 硬件加速 | 支持 AES-NI / ARM Crypto Extensions |

### 2.2 非对称加密 —— RSA（Rivest–Shamir–Adleman）

#### 2.2.1 算法思想

RSA 的安全性基于**大整数分解的计算困难性**：给定两个大素数 $p, q$ 的乘积 $n = p \cdot q$，在不知道 $p$ 和 $q$ 的情况下，从 $n$ 求出 $p$ 和 $q$ 在计算上是不可行的（当 $n$ 足够大时）。

RSA 加密数学基础 —— **欧拉定理**：

$$m^{ed} \equiv m \pmod{n}, \quad \text{当 } ed \equiv 1 \pmod{\phi(n)}$$

其中 $\phi(n) = (p-1)(q-1)$ 为欧拉函数。加密计算 $c = m^e \bmod n$，解密计算 $m = c^d \bmod n$。

#### 2.2.2 为什么要用 OAEP 填充？

**教科书式 RSA**（直接 $c = m^e \bmod n$）存在多种攻击：

| 攻击类型 | 描述 |
|----------|------|
| 确定性加密 | 相同明文总是产生相同密文，易受统计分析 |
| 选择密文攻击（CCA） | 攻击者可构造特殊密文，利用解密预言机恢复明文 |
| 小指数攻击 | 当 $e=3$ 且 $m^3 < n$ 时，直接开立方即可恢复明文 |
| Coppersmith 攻击 | 已知部分明文时可在多项式时间内恢复完整明文 |

OAEP（Optimal Asymmetric Encryption Padding）是 Bellare 和 Rogaway 在 1994 年提出的填充方案，通过**随机填充 + 两轮 Feistel 网络**将明文盲化，使 RSA-OAEP 在随机预言机模型下达到 **IND-CCA2** 安全。

#### 2.2.3 参数选择

| 参数 | 取值 | 理由 |
|------|------|------|
| 密钥长度 $n$ | 2048 bits | NIST SP 800-57 推荐，安全强度 ≈ 112 bits |
| 公钥指数 $e$ | 65537 ($2^{16}+1$) | 质数，二进制仅两个 1，加密速度快 |
| Hash 函数 | SHA-256 | 碰撞安全，MGF1 掩码生成函数基于此 |
| 明文上限 | ~190 bytes | $n$ 长度减去两次 hash 输出和两个填充字节 |

#### 2.2.4 算法步骤

```
密钥生成:
  1. 随机选择两个大素数 p, q（各 ~1024 bits）
  2. 计算 n = p × q
  3. 计算 φ(n) = (p-1)(q-1)
  4. 选择 e = 65537，满足 gcd(e, φ(n)) = 1
  5. 计算 d ≡ e⁻¹ (mod φ(n))
  6. 公钥 = (n, e)，私钥 = (n, d)

RSA-OAEP 加密:
  输入: 明文 m, 公钥 (n, e)
  输出: 密文 c
  1. 对 m 进行 OAEP 填充（随机种子 + SHA-256 + MGF1）
  2. c = EM^e mod n（EM 为填充后的编码消息）

RSA-OAEP 解密:
  输入: 密文 c, 私钥 (n, d)
  输出: 明文 m
  1. EM = c^d mod n
  2. 验证 OAEP 填充结构，提取 m
  3. 若填充校验失败，拒绝解密（防 CCA）
```

#### 2.2.5 算法复杂度

| 指标 | RSA-2048 |
|------|----------|
| 密钥生成复杂度 | $O(k^3) \sim O(k^4)$，k 为密钥长度（涉及大素数生成） |
| 加密复杂度 | $O(k^2 \log e)$，e = 65537 时仅需 17 次模乘 |
| 解密复杂度 | $O(k^2 \log d) \approx O(k^3)$，比加密慢 10~100 倍 |
| 密文膨胀 | 明文 190 bytes → 密文 256 bytes（约 1.35×） |
| 明文上限 | $\lfloor (k/8) - 2h/8 - 2 \rfloor$ bytes |

### 2.3 混合加密 —— RSA 密钥封装 + AES 数据加密

#### 2.3.1 设计动机

对称加密和非对称加密各有优劣：

| 维度 | 对称加密（AES） | 非对称加密（RSA） |
|------|-----------------|-------------------|
| **加密速度** | 极快（GB/s 级别） | 慢（次/秒 级别） |
| **密钥分发** | 需安全信道 | 公钥可公开分发 |
| **数据量限制** | 无限制 | 约 190 bytes（RSA-2048） |
| **计算成本** | 低 | 高 |

**混合加密 = 取两者的长处**：用 RSA 安全地传递一次性会话密钥（密钥封装），用 AES 高效地加密实际数据（数据封装）。

#### 2.3.2 GCM 模式选择理由

本实验数据封装采用 **AES-256-GCM**（Galois/Counter Mode），这是一种 **AEAD**（带关联数据的认证加密）模式：

- **机密性**：CTR 模式加密，可并行
- **完整性 + 认证性**：GHASH 多项式认证标签，防篡改
- **无需额外 HMAC**：CBC 模式需额外 HMAC 提供认证，GCM 二合一
- **仅需加密原语**：GCM 只用加密方向的 AES（解密也用加密），实现更简单

**GCM 加密**（简化）：
$$C = P \oplus \text{AES}_K(\text{Counter}), \quad T = \text{GHASH}_H(A, C) \oplus \text{AES}_K(\text{Nonce} \| 0^{31}1)$$

**GCM 解密**先验证 Tag $T$：若 Tag 不匹配则拒绝解密（防篡改）；验证通过后以相同 CTR 流异或恢复明文。

#### 2.3.3 协议流程

```
┌─────────────────────────┐              ┌─────────────────────────┐
│         Alice            │              │          Bob            │
│     （消息接收方）        │              │      （消息发送方）       │
├─────────────────────────┤              ├─────────────────────────┤
│                         │    公钥      │                         │
│ 1. 生成 RSA-2048 密钥对  │────────────▶│ 2. 获取 Alice 的公钥     │
│    - 私钥: sk_A (保密)   │              │                         │
│    - 公钥: pk_A (发布)   │              │ 3. 随机生成 AES-256     │
│                         │              │    会话密钥 K_s（一次性）│
│                         │              │                         │
│                         │              │ 4. AES-256-GCM 加密明文 │
│                         │              │    C, T = AES_GCM(K_s,  │
│                         │              │           nonce, P)     │
│                         │              │                         │
│                         │              │ 5. RSA-OAEP 加密 K_s    │
│                         │              │    E_K = RSA_Enc(pk_A,  │
│                         │              │           K_s)          │
│                         │              │                         │
│                         │   密文包     │ 6. 发送:                │
│                         │◀────────────│    (E_K, nonce, C)      │
│                         │              │                         │
│ 7. RSA 解密:             │              │                         │
│    K_s = RSA_Dec(sk_A,  │              │                         │
│          E_K)           │              │                         │
│                         │              │                         │
│ 8. AES-GCM 解密:         │              │                         │
│    P = AES_GCM_Dec(K_s, │              │                         │
│         nonce, C)       │              │                         │
│    Tag 自动验证          │              │                         │
└─────────────────────────┘              └─────────────────────────┘
```

#### 2.3.4 安全性分析

| 安全属性 | 实现方式 |
|----------|----------|
| **机密性** | AES-256-GCM 加密数据；RSA-2048 保护会话密钥 |
| **完整性** | GCM 128-bit 认证标签，任何篡改都会被检测 |
| **认证性** | GCM Tag 确认为持有正确密钥的发送方所发 |
| **抗重放** | 每次会话使用新的随机 nonce + 一次性会话密钥 |
| **IND-CCA2** | RSA-OAEP 在随机预言机模型下达到 |
| **前向安全性** | ⚠️ 本实验未实现，需 DH 密钥交换（如 ECDHE） |

### 4.4 关键代码实现

#### 4.4.1 AES-CBC 加密核心代码

```python
# AES-256-CBC 加密
def aes_encrypt(plaintext: str, key: bytes) -> tuple[bytes, bytes]:
    iv = os.urandom(16)                    # ① 128-bit 随机 IV
    cipher = Cipher(algorithms.AES(key), modes.CBC(iv))
    encryptor = cipher.encryptor()

    padder = padding.PKCS7(128).padder()   # ② PKCS7 填充
    padded_data = padder.update(plaintext.encode("utf-8")) + padder.finalize()

    ciphertext = encryptor.update(padded_data) + encryptor.finalize()  # ③ 加密
    return iv, ciphertext
```

**设计要点**：
1. IV 由 `os.urandom(16)` 生成（密码学安全随机源），每次加密使用不同 IV
2. PKCS7 填充保证明文长度是 AES 块大小（128 bits）的整数倍
3. `cryptography` 库的加密器为流式接口，支持分块输入

#### 4.4.2 RSA-OAEP 解密核心代码

```python
def rsa_decrypt(ciphertext: bytes, private_key) -> str:
    plaintext_bytes = private_key.decrypt(
        ciphertext,
        padding.OAEP(
            mgf=padding.MGF1(algorithm=hashes.SHA256()),  # 掩码生成函数
            algorithm=hashes.SHA256(),                      # 哈希函数
            label=None,                                     # 可选标签
        ),
    )
    return plaintext_bytes.decode("utf-8")
```

**设计要点**：OAEP 的 MGF1 和 Hash 都使用 SHA-256，确保 128-bit 安全级别与 RSA-2048 匹配。任何填充校验失败都会抛出异常（防 CCA）。

#### 4.4.3 混合加密核心代码（Bob 加密）

```python
def encrypt_hybrid_message(self, plaintext: str, alice_public_key) -> dict:
    # ① 生成一次性 AES-256 会话密钥
    aes_key = AESGCM.generate_key(bit_length=256)

    # ② AES-256-GCM 加密数据（数据封装）
    aesgcm = AESGCM(aes_key)
    nonce = os.urandom(12)          # GCM 推荐 96-bit nonce
    ciphertext = aesgcm.encrypt(nonce, plaintext.encode("utf-8"), None)
    # ciphertext 末尾 16 bytes 为 GCM 认证标签（自动附加）

    # ③ RSA-OAEP 加密 AES 会话密钥（密钥封装）
    encrypted_key = alice_public_key.encrypt(
        aes_key,
        padding.OAEP(mgf=padding.MGF1(hashes.SHA256()),
                      algorithm=hashes.SHA256(), label=None),
    )

    return {"encrypted_key": encrypted_key,
            "nonce": nonce,
            "ciphertext": ciphertext}
```

**设计要点**：
- AES 会话密钥由 `AESGCM.generate_key(256)` 生成（一次性随机密钥）
- GCM `encrypt()` 自动将 16-byte 认证标签附加到密文末尾，无需手动处理
- `nonce` 使用 12 bytes（96 bits），是 GCM 的最优长度
- 函数返回的字典即"密文包"，通过不安全信道发送给 Alice

### 5.1 实验一：AES-256-CBC 对称加解密

#### 测试用例 1：常规中英文混合消息

| 项目 | 内容 |
|------|------|
| **输入明文** | `Hello, Cryptography! 这是一条使用AES加密的测试消息。` |
| **密钥** | `0b0d02986ae15f6bb1545489ebb8d3dbdfa1bba2c7e0194298fdafb632ee99a9`（256 bits） |
| **IV** | `4b98c00ed955a24012cd4b92f6849917`（128 bits） |
| **密文** | `bd281283b5c15f207f3ef0c0ea7f710d...`（80 bytes） |
| **解密结果** | `Hello, Cryptography! 这是一条使用AES加密的测试消息。` ✅ |
| **验证** | 解密结果与原文完全一致 |

**分析**：明文 58 bytes，经过 PKCS7 填充后为 64 bytes，加密为 80 bytes（64 明文块 + 16 IV）。密文完全随机，无法从中推断任何明文信息。

#### 测试用例 2：错误密钥解密

| 项目 | 内容 |
|------|------|
| **操作** | 使用不同于加密时的随机密钥尝试解密密文 |
| **结果** | ❌ 解密失败（引发异常） |
| **原因** | 解密后 PKCS7 填充校验失败——解密出了乱码数据 |

**安全含义**：即使攻击者获得密文，没有正确密钥也无法解密。PKCS7 填充的格式校验充当了隐式的完整性检查（但不应依赖于此，应使用 GCM 等认证加密）。

#### 相同明文两次加密对比

| | 第一次加密 | 第二次加密 |
|------|------------|------------|
| **IV** | `4b98c00e...` | `a3f1e278...`（完全不同） |
| **密文前 8 bytes** | `bd281283b5c15f20` | `7e9a4c3f21d8ab90`（完全不同） |

**分析**：随机 IV 使同一明文每次加密产生不同密文，**语义安全**得到保证。

### 5.2 实验二：RSA-2048 非对称加解密

#### 测试用例

| 项目 | 内容 |
|------|------|
| **明文** | `RSA实验消息: 使用2048-bit密钥和OAEP填充。`（53 bytes） |
| **密钥长度** | 2048 bits |
| **模数 n 长度** | 2048 bits |
| **公钥指数 e** | 65537 |
| **密文长度** | 256 bytes（= 2048 bits） |
| **解密结果** | `RSA实验消息: 使用2048-bit密钥和OAEP填充。` ✅ |
| **验证** | 解密结果与原文完全一致 |

**关键观察**：
- 53 bytes 明文 → 256 bytes 密文，密文膨胀比约 **4.8×**
- 密文长度恒为 256 bytes（与 RSA 密钥长度一致），无论明文多短
- 这解释了为什么 RSA 不适合直接加密大量数据——效率低且密文膨胀严重

#### RSA 单次加密最大明文长度

对 RSA-2048 + OAEP(SHA-256)：$2048/8 - 2 \times 256/8 - 2 = 256 - 64 - 2 = 190$ bytes。

超过此限制时，需使用混合加密（见实验三）。

### 5.3 实验三：混合加密（RSA + AES-GCM）

#### 测试用例

| 项目 | 内容 |
|------|------|
| **明文** | `这是一条通过混合加密传输的机密消息。\nRSA 负责安全分发密钥，AES 负责高效加密数据。\n这种方式结合了非对称加密的安全性和对称加密的高效性。`（196 bytes） |
| **AES 会话密钥** | 256 bits（一次性，随机生成） |
| **GCM Nonce** | 96 bits（随机生成） |
| **密文** | 212 bytes（含 16-byte 认证标签） |
| **RSA 加密的密钥** | 256 bytes |
| **传输总量** | 480 bytes |
| **解密结果** | 与原文完全一致 ✅ |

#### 传输效率分析

| 指标 | 数值 |
|------|------|
| 有效载荷（明文） | 196 bytes |
| 总密文传输量 | 480 bytes |
| 总膨胀比 | 480 / 196 ≈ **2.45×** |

> **如果用纯 RSA 加密 196 bytes**：需要分割为 2 段分别加密 = 512 bytes（2 × 256）。混合加密仅需 480 bytes，且对于更大的数据量，混合加密的优势会更显著（RSA 部分的 256 bytes 固定开销被摊薄）。

### 5.4 各方案对比总结

| 方案 | 适用数据量 | 密文膨胀 | 速度 | 密钥分发 | 完整性保护 |
|------|-----------|----------|------|----------|------------|
| AES-256-CBC | 无限制 | ~1× + 16B IV | 极快 | ❌ 需安全信道 | ❌ 无认证 |
| RSA-2048-OAEP | ≤ 190 bytes | ~4~5× | 慢 | ✅ 公钥可公开 | ❌ 无认证 |
| RSA + AES-GCM（混合） | 无限制 | ~1× + 256B 密钥 | 快 | ✅ 公钥可公开 | ✅ GCM 认证 |

### 5.5 遇到的问题与解决方案

| 序号 | 问题 | 原因 | 解决方案 |
|------|------|------|----------|
| 1 | `hashes.Hash().update().finalize()` 返回 `NoneType has no attribute 'finalize'` | `update()` 方法修改对象自身但返回 `None`，不支持链式调用 | 分三步调用：创建 Hash 对象 → `update()` → `finalize()` |
| 2 | `pip install cryptography` 报 "externally-managed-environment" 错误 | macOS Homebrew Python 3 启用了 PEP 668 外部环境管理 | 无需安装——系统已预装 `cryptography` v48.0.0 |
| 3 | RSA-OAEP 加密数据超长时报错 | RSA-2048 + OAEP(SHA-256) 最多加密约 190 bytes | 在代码注释中明确容量限制，混合加密实验用 AES 处理大数据 |
| 4 | GCM nonce 长度选择 | GCM 标准允许 1~2⁶⁴-1 bytes，但 96 bits 是性能最优长度 | 使用 `os.urandom(12)` 生成 96-bit nonce |

## 六、实验总结

### 6.1 主要收获

1. **AES 实现层面**：掌握了 AES-CBC 模式的完整加解密流程，理解了 IV 的必要性（语义安全）、PKCS7 填充机制，以及为什么生产环境推荐 GCM 等 AEAD 模式替代 CBC。

2. **RSA 实现层面**：深入理解了 RSA 的数学本质（模幂运算 + 大整数分解难题），认识到教科书式 RSA 的安全弱点以及 OAEP 填充为何必不可少。

3. **混合加密设计层面**：这是本次实验最核心的收获——密码学工程实践中几乎不会单独使用某一种加密原语。混合加密体现了 **"各取所长"** 的设计哲学：用 RSA 解决密钥分发难题（单次 256 bytes 开销可接受），用 AES-GCM 承担数据加密主体（速度快、无膨胀），用 GCM 的 AEAD 属性一次性解决机密性 + 完整性。

4. **安全思维**：体会到密码学中 "自创算法 = 不安全" 的铁律——直接使用教科书式 RSA 加密明文会引入多种攻击向量，CBC 模式不加 HMAC 无法保证完整性。正确的做法是使用经过密码学界充分分析的标准化构造（OAEP、GCM）。

### 6.2 本实验的局限与改进方向

| 局限 | 改进方向 |
|------|----------|
| 未实现前向安全性 | 引入 ECDHE 密钥交换，每次会话生成临时密钥对 |
| 未涉及密钥管理 | 引入 PKI 证书体系（X.509）、CA 信任链 |
| 会话密钥仅一次性使用 | 可实现会话密钥派生（HKDF）+ 密钥轮换 |
| 未实现身份认证 | 引入数字签名（RSA-PSS / ECDSA），Alice 签名挑战值，Bob 验证 |
| 未涉及侧信道防御 | RSA 解密需使用恒定时间实现（`cryptography` 库已内置） |

### 6.3 心得

密码学实验最大的感受是：**安全的实现远比正确性验证复杂**。所有的三个实验都能正常加解密、输出 "✅ 验证通过"，但这并不意味系统安全。从密钥长度选择、填充方案取舍、到协议流程设计，每一个环节的疏忽都可能成为攻击者的突破口。

举例来说：如果我在实验三中使用 AES-CBC 替代 AES-GCM（仅因为 CBC "更熟悉"），就丢失了完整性保护——攻击者可篡改密文而接收方毫无察觉。这正是为什么密码工程领域反复强调：**不要自己发明密码学方案**，应使用标准化协议（TLS 1.3、Noise Protocol 等）。本实验作为学习用途，最大的价值在于通过亲手实现来理解这些标准化协议背后 "为什么这样设计"。
